## 1. ROOT CAUSE

The insert into `clients` uses column names that **do not exist** in the current Supabase schema:

- **First error (fixed):** `client_type` — removed from insert.
- **Second error (current):** `tax_id` — still in insert; PGRST204 "Could not find the 'tax_id' column of 'clients' in the schema cache".

So the **root cause** is: the app assumes a schema (e.g. `company_name`, `tax_id`, `client_type`, …) that does not match the real `clients` table. There are no migrations or schema files in the repo to confirm the real columns; inference is from **working read paths** and **runtime errors** only.

## 8. MINIMAL SAFE INSERT PAYLOAD FOR `clients`

- **Avoid** any column that is confirmed invalid: `name`, `client_type`, `tax_id`.
- **Use** only columns we have positive evidence for, or that are standard (e.g. `id` is generated).

Evidence:

- `id`: generated by DB.
- `created_at`: exists (order works); typically defaulted (e.g. `now()`), so no need to send.
- Name: exactly one of `company_name` or `client_name` exists; we don’t know which from code.

**Minimal safe insert (evidence-based):**

- **Option A (try first):**  
  `{ company_name: payload.name }`  
  Only one column; if the real column is `company_name`, insert succeeds.
- **If PGRST204 for `company_name`:**  
  **Option B:**  
  `{ client_name: payload.name }`  
  So the real name column is `client_name`.

Do **not** include `tax_id`, `client_type`, `contact_person`, `phone`, `email`, `notes` or `name` in the insert until the schema is confirmed or extended.
